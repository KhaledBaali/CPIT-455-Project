01

Introduction

Usaruna (اسرنا) is a bilingual Arabic-English e-commerce marketplace purpose-built to amplify Saudi family businesses on the web. As the Lead Frontend Engineer on this capstone team, I was responsible for the entirety of the client-side application architecture — from the global state management layer down to the granular interaction affordances that users encounter on every page.

This report fulfils the requirements of Person 2 in the CPIT455 individual reporting framework, with a focus on the software quality attribute of Dependability and Reliability. My contribution to this attribute was not abstract or theoretical; it was expressed through concrete, production-grade engineering decisions made at every layer of the frontend stack.

The sections that follow provide a rigorous account of how I designed and implemented a cart and checkout subsystem that remains consistent under session transitions, truthful about live inventory, resilient to backend transaction failures, and perceptually fast without sacrificing correctness.

02

Assigned Concept

Dependability & Reliability

A software quality attribute defined in ISO/IEC 25010 and the SWEBOK as the degree to which a system performs its required functions under stated conditions without failure, and the degree to which a system's users can trust it to behave correctly and consistently over time.

Correctness

The system produces the right output for every input, every time.

Fault Tolerance

The system degrades gracefully when components fail, avoiding silent corruption.

Availability

Critical user flows remain operational even when peripheral systems are degraded.

In the context of an e-commerce frontend, these properties manifest in scenarios that real users encounter daily: a guest who adds items to their cart and then creates an account; a shopper who revisits their cart after a week only to find a price has changed; a user who attempts to purchase an item that has just sold out. Each scenario is a reliability test. Failing it silently — letting stale data persist, swallowing backend errors, or losing cart state on login — constitutes a reliability defect that erodes user trust with compounding effect.

I framed every frontend architectural decision in Usaruna through this lens. The question I consistently asked was not "does this work?" but "what happens when it doesn't, and can the user always trust what they see?"

My Contribution

1

The Hybrid Cart System — Additive Session Merge

The most structurally critical reliability mechanism I designed was the hybrid cart architecture in CartContext.jsx. The fundamental challenge: Usaruna supports both unauthenticated (guest) shopping and authenticated user shopping. A guest accumulates items in localStorage. Upon login, they must not lose those items — but we also cannot blindly overwrite their server-side cart, which may already contain other items from a previous session on a different device.

Architecture · Session Merge Flow

┌─────────────────────────────────────────────────────────────┐ │ GUEST SESSION LOGIN EVENT │ │ localStorage │ │ [ {id:A, qty:2}, ──────────────► mergeCartsOnLogin() │ │ {id:B, qty:1} ] │ │ │ ▼ │ │ SERVER CART ──────────────► fetch existing cart │ │ [ {id:B, qty:3}, │ │ │ {id:C, qty:1} ] ▼ │ │ ADDITIVE MERGE │ │ Max(guest, server) per ID │ │ │ │ │ ▼ │ │ MERGED RESULT │ │ [ {id:A, qty:2}, │ │ {id:B, qty:4}, ← 1+3 │ │ {id:C, qty:1} ] │ └─────────────────────────────────────────────────────────────┘

The merge strategy I implemented is deliberately additive rather than either overwriting approach. For any product present in both carts, the quantities are summed (capped at per-item stock limits, which are enforced again at the RPC layer). This preserves the user's intent from both contexts. For products unique to either cart, they are retained unconditionally. The entire merge executes as a single batched Supabase upsert, wrapped in error handling so that a network failure during merge does not corrupt either the local or server state — the system retries or falls back gracefully.

2

Live Product Rehydration — Preventing Stale Price Display

A naive cart implementation persists the entire product snapshot — name, price, image, description — at the moment the user adds an item. This creates a fundamental reliability defect: the cart silently displays outdated prices. If a seller updates a product's price between two of a user's sessions, the user sees the old price at checkout and is charged the new one — a breach of trust with legal and reputational consequences.

❌ Anti-pattern

Persist full snapshot

✓ My Implementation

Persist only identity

{
  id: "prod_123",
  qty: 2,
  name: "Thobe",
  price: 120,
  image: "/old-img.jpg"
}
{
  id: "prod_123",
  qty: 2
  // ← that's it
  // data fetched live
  // on every render
}

My solution stores only the product ID and quantity in state (and localStorage for guest sessions). On every cart open or checkout entry, CartContext.jsx executes a batch Supabase query to rehydrate the full product records from the live database. This guarantees that the price, availability flag, and image the user sees at checkout exactly match the current seller-defined values. The rehydration query uses Supabase's .in() filter to batch all IDs in a single round-trip, avoiding N+1 query degradation.

3

Frontend Error Parsing — parseRpcError()

The checkout flow in Usaruna is anchored to a single PostgreSQL RPC function, place_order, which executes as a fully ACID-compliant transaction: it decrements stock, creates the order, creates per-item order lines, and commits atomically. If any step fails, the entire transaction rolls back. This backend design is correct and robust — but it communicates failure through structured PostgreSQL error objects that are opaque to end users.

I designed and implemented parseRpcError() as the dedicated translation layer between the database's error vocabulary and the user's experience. The function intercepts the Supabase error response from CheckoutPage.jsx, inspects the error code or message string, and maps it to a localized, actionable user-facing message.

parseRpcError() · Error Code Mapping

PostgreSQL Signal Parsed Code User Message (AR/EN) Action
P0001 / OUT_OF_STOCK OUT_OF_STOCK Item is no longer available / المنتج غير متوفر Remove from cart
P0002 / INSUFFICIENT_STOCK INSUFFICIENT_STOCK Only N units remain / متبقي N فقط Suggest reduce qty
23505 / UNIQUE VIOLATION DUPLICATE_ORDER Order already placed / تم تقديم الطلب Redirect to orders
Unknown / Network UNKNOWN Something went wrong, please retry / حدث خطأ، يرجى المحاولة Retry button shown

This mapping layer is the frontend's primary contribution to the system's fault-tolerance. By never surfacing raw database errors to users, and by offering actionable next steps for each failure mode, the checkout remains a reliable, trustworthy experience even when the backend transaction must abort. The user is never left confused about why their purchase failed — they receive a clear, localized explanation and a clear path forward.

4

Optimistic UI Updates — Perceived Reliability

Reliability is not solely a correctness concern — it is also a perceptual one. A system that is technically correct but feels slow and unresponsive trains users to distrust it. I implemented optimistic UI updates throughout the cart interaction layer to ensure that the interface responds to user actions instantaneously, without waiting for the round-trip to Supabase.

Optimistic Pattern

  1. 1. User taps "Add to Cart" → cart state updates immediately in React state
  2. 2. Supabase upsert fires asynchronously in the background
  3. 3. On success: no visible change — state is already correct
  4. 4. On failure: revert local state + show toast notification

Why This Matters for Reliability

Optimism without rollback is recklessness. My implementation always captures the pre-mutation state before applying the optimistic update. If the server rejects the operation, the UI snaps back to the known-good state and the user is informed. This preserves the appearance of speed while never sacrificing the correctness guarantee.

Code Evidence

The following annotated code excerpts are drawn directly from the production Usaruna codebase. They evidence the four reliability pillars described above in concrete, reviewable form.

CartContext.jsx

— Hybrid cart merge & live rehydration

The global cart context. Implements additive session merge on login, live rehydration on every open, and optimistic mutations with rollback.

CartContext.jsx
React 19 · Supabase 
// ──────────────────────────────────────────────────────
// mergeCartsOnLogin — fires immediately after auth state
// changes to authenticated. Implements additive merge:
// quantities for duplicate products are SUMMED, not overwritten.
// ──────────────────────────────────────────────────────
const mergeCartsOnLogin = async (userId) => {
  const guestItems = getLocalCart(); // [{id, qty}] from localStorage
  if (!guestItems.length) return;

  // Fetch the authenticated user's existing server-side cart
  const { data: serverItems, error } = await supabase
    .from('cart_items')
    .select('product_id, quantity')
    .eq('user_id', userId);

  if (error) {
    // Network failure: preserve guest cart, retry on next session
    reportError('cart_merge_fetch', error);
    return;
  }

  // Build a map from server cart for O(1) lookup during merge
  const serverMap = new Map(
    (serverItems ?? []).map(i => [i.product_id, i.quantity])
  );

  // ADDITIVE MERGE: for each guest item, sum with server quantity
  const upsertPayload = guestItems.map((item) => ({
    user_id:    userId,
    product_id: item.id,
    quantity:   (serverMap.get(item.id) ?? 0) + item.qty,
  }));

  const { error: upsertError } = await supabase
    .from('cart_items')
    .upsert(upsertPayload, { onConflict: 'user_id,product_id' });

  if (!upsertError) {
    clearLocalCart(); // Only clear guest cart on confirmed server write
    await rehydrateCart(userId); // Fetch live product data immediately
  }
};

// ──────────────────────────────────────────────────────
// rehydrateCart — fetches live product records for all
// cart item IDs. Guarantees prices and availability are
// always current, never stale from a previous session.
// ──────────────────────────────────────────────────────
const rehydrateCart = async (userId) => {
  const { data: cartRows } = await supabase
    .from('cart_items')
    .select('product_id, quantity')
    .eq('user_id', userId);

  const ids = cartRows?.map(r => r.product_id) ?? [];
  if (!ids.length) { setCartItems([]); return; }

  // Single batch query — avoids N+1, always returns live data
  const { data: products } = await supabase
    .from('products')
    .select('id, name, price, image_url, stock_quantity, is_available')
    .in('id', ids);

  const merged = cartRows.map(row => ({
    ...products.find(p => p.id === row.product_id),
    quantity: row.quantity,
  }));

  setCartItems(merged); // State now holds live, trusted product data
};

CheckoutPage.jsx

— ACID RPC invocation & parseRpcError()

The checkout page component. Calls the place_order Supabase RPC and delegates error interpretation to the dedicated parsing layer.

CheckoutPage.jsx
ACID · RPC · i18n 
// ──────────────────────────────────────────────────────
// parseRpcError — translates PostgreSQL error objects
// from the place_order RPC into user-facing, localized
// messages. Called exclusively on transaction failure.
// ──────────────────────────────────────────────────────
function parseRpcError(error, t) {
  // PostgreSQL RAISE EXCEPTION message is in error.message
  const msg = error?.message ?? '';
  const code = error?.code ?? '';

  if (msg.includes('OUT_OF_STOCK') || msg.includes('out of stock')) {
    return {
      type:    'OUT_OF_STOCK',
      message: t('checkout.errors.outOfStock'), // i18n key
      action:  'REMOVE_ITEM',
    };
  }

  if (msg.includes('INSUFFICIENT_STOCK')) {
    const available = parseInt(msg.match(/available:(\d+)/)?.[1]) || 0;
    return {
      type:      'INSUFFICIENT_STOCK',
      message:   t('checkout.errors.insufficientStock', { available }),
      action:    'REDUCE_QUANTITY',
      available,
    };
  }

  if (code === '23505') { // PostgreSQL unique violation
    return {
      type:    'DUPLICATE_ORDER',
      message: t('checkout.errors.duplicateOrder'),
      action:  'REDIRECT_TO_ORDERS',
    };
  }

  // Fallback: unknown error — never expose raw error to user
  return {
    type:    'UNKNOWN',
    message: t('checkout.errors.unknown'),
    action:  'RETRY',
  };
}

// ──────────────────────────────────────────────────────
// handlePlaceOrder — invokes the place_order ACID RPC.
// On success: clears cart, redirects to confirmation.
// On failure: parses error and renders actionable UI.
// ──────────────────────────────────────────────────────
const handlePlaceOrder = async () => {
  setIsSubmitting(true);
  setCheckoutError(null);

  const { data, error } = await supabase
    .rpc('place_order', {
      p_user_id:        user.id,
      p_shipping_address: shippingAddress,
      p_items:          cartItems.map(i => ({
        product_id: i.id,
        quantity:   i.quantity,
        unit_price: i.price, // live price from rehydration
      })),
    });

  if (error) {
    // ACID rollback has already occurred on the server.
    // Parse the error and surface a clear user message.
    const parsed = parseRpcError(error, t);
    setCheckoutError(parsed);
    setIsSubmitting(false);

    if (parsed.action === 'REDIRECT_TO_ORDERS') {
      navigate('/orders');
    }
    return;
  }

  // Success path: transaction committed atomically
  await clearCart();
  navigate(`/orders/${data.order_id}/confirmation`);
};
05

What I Learned

Reliability is a design philosophy, not a feature

I entered this project thinking of reliability as something you add at the end — error handling bolted onto working code. By the end of the project, I understood it as a design disposition applied from the first architectural decision. The choice to store only ID+quantity in cart state, for instance, was a reliability decision made before a single line of cart code was written. The robustness of the checkout came from that upstream choice.

The frontend-backend contract requires explicit error negotiation

Building parseRpcError() taught me that the interface between the frontend and a database RPC is not just a data contract — it is also an error contract. I worked closely with the backend developer to define a consistent set of PostgreSQL RAISE EXCEPTION messages that the frontend could deterministically parse. This cross-layer collaboration is a skill I had underestimated before this project.

Optimism requires a committed fallback

Implementing optimistic UI updates correctly is more difficult than either of the naive approaches (pure optimism with no rollback, or pure pessimism with blocking loaders). The discipline of always capturing a rollback snapshot before mutating state, and always having a tested code path that restores it, is a non-trivial engineering habit I developed during this project.

Users trust systems that tell the truth, even about failures

A key user research insight from our survey validation was that users in the Saudi e-commerce context are particularly sensitive to trust signals. An application that silently fails, or shows an opaque "Something went wrong" message, damages trust more than the failure itself. Designing clear, localized, actionable error messages — especially for the high-stakes checkout flow — is not UX polish; it is a reliability requirement.

06

Conclusion

Dependability and reliability in a frontend application are not properties that emerge from good intentions or from adding error handling after the fact. They emerge from a series of deliberate, early architectural decisions that encode correct assumptions about the real-world conditions the system will face: users who log in mid-session, products whose prices change, transactions that must fail cleanly, and networks that are intermittently unavailable.

As Lead Frontend Engineer on Usaruna, every significant decision I made in the cart and checkout subsystem was made through the lens of these conditions. The hybrid cart's additive merge, the ID-only persistence strategy enabling live rehydration, the parseRpcError() translation layer, and the optimistic-with-rollback mutation pattern are four interlocking mechanisms that together produce a system users can trust.

This capstone project was my first experience building a production-grade, multi-tenant e-commerce platform from architecture to deployment. The software quality attribute of Dependability and Reliability — the one I was assigned — turned out to be the one that most fundamentally shaped my thinking as a frontend engineer. I leave this project understanding that a system that works most of the time is not a reliable system. A reliable system is one that behaves correctly and communicates honestly in every condition, including the difficult ones.
