Person 1 Software Reuse

Busam Barak

Software Architect & Dependency Strategist

Concept Definition

Software Reuse is the practice of leveraging existing software assets — libraries, frameworks, services, and patterns — to reduce development effort, increase reliability, and accelerate delivery. Effective reuse requires disciplined dependency selection, version management, and understanding the contracts each reused component provides.

Key Contributions

  • React 19 Component Architecture

    Structured the entire UI as a tree of reusable, composable components, eliminating duplicate UI logic across buyer and seller flows.

  • Supabase BaaS — Backend Without Code

    Replaced ~3,000 lines of custom backend (auth, REST API, file storage) by adopting Supabase as a Backend-as-a-Service layer, reusing its PostgREST auto-generated API.

  • Vite Build Toolchain

    Adopted Vite's pre-configured HMR pipeline, eliminating custom Webpack configuration and reusing community-maintained presets.

  • Tailwind CSS Design Tokens

    Reused Tailwind's utility-first token system for spacing, typography, and color — the entire design system with zero custom CSS files.

Person 2 Dependability & Reliability

Khaled Baali

Lead Frontend Engineer

Concept Definition

Dependability & Reliability is the degree to which a system performs its required functions correctly and consistently under stated conditions. In an e-commerce frontend it manifests as cart state that survives session transitions, prices that are always live, and failures that are communicated honestly and actionably to the user.

Key Contributions

  • Hybrid Cart — Additive Session Merge

    On login, guest localStorage cart is additively merged with the server cart without data loss from either context.

  • Live Product Rehydration

    Only product ID + quantity is persisted; full product data (price, availability) is fetched live on every cart open, preventing stale price display.

  • parseRpcError() — Fault Translation Layer

    Intercepts PostgreSQL ACID transaction errors (OUT_OF_STOCK, INSUFFICIENT_STOCK) and surfaces localized, actionable messages.

  • Optimistic UI with Rollback

    Cart mutations apply instantly to state; a pre-mutation snapshot is captured and restored atomically if the Supabase write fails.

Person 3 Security & Access Control

Rami Jameel

Security Engineer & Database Policy Architect

Concept Definition

Security & Access Control encompasses the policies, mechanisms, and design patterns that ensure users can only read and write data they are authorized to access. In a multi-vendor marketplace, this is the critical engineering discipline that prevents one seller from viewing or modifying another seller's data.

Key Contributions

  • Supabase Auth Integration

    Implemented JWT-based authentication with Supabase Auth, ensuring every API request carries a signed user identity token verified server-side.

  • Row-Level Security (RLS) Policies

    Designed and deployed PostgreSQL RLS policies on all sensitive tables (products, orders, cart_items), enforcing multi-vendor isolation at the database layer.

  • SECURITY DEFINER Guard on RPC

    The place_order RPC runs as SECURITY DEFINER, granting it elevated privileges only for the exact operations required, scoped by explicit auth.uid() checks.

  • Frontend Route Guards

    React Router protected routes prevent unauthenticated users from accessing seller dashboards, checkout flows, and order management pages.

Evidence · RLS Policies (SQL)

-- Sellers can only manage their own products
CREATE POLICY "seller_owns_products"
  ON products
  FOR ALL
  USING (auth.uid() = seller_id)
  WITH CHECK (auth.uid() = seller_id);
 
-- Buyers see only their own cart items
CREATE POLICY "user_owns_cart"
  ON cart_items
  FOR ALL
  USING (auth.uid() = user_id);
 
-- RPC elevated scope, user-scoped guard
CREATE FUNCTION place_order(...)
  SECURITY DEFINER
  SET search_path = public AS $$
  ...
  WHERE user_id = auth.uid() -- never trust input
$$;
Person 4 Distributed & Cloud Architecture

Ahmed Aljohani

Cloud Infrastructure & Architecture Lead

Concept Definition

Distributed & Cloud Architecture is the design discipline of decomposing a system into independently deployable, scalable services hosted on managed cloud infrastructure. The key concerns are the clear separation of responsibilities across layers, minimizing operational burden, and achieving horizontal scalability without custom infrastructure management.

Key Contributions

  • BaaS Architectural Model

    Designed the overall system as a Backend-as-a-Service architecture: the React SPA is a thin, stateless presentation layer; all data, auth, and file logic lives in Supabase's managed cloud.

  • Layer Separation: UI ↔ Data ↔ Storage

    Enforced strict separation: React handles rendering, Supabase PostgREST handles CRUD queries, Supabase Storage handles product images — each replaceable independently.

  • PostgreSQL as Distributed State Store

    Designed the database schema as the single authoritative source of truth for cart state, order lifecycle, and product inventory — enabling multi-device consistency.

  • City-Based Geo-Filtering

    Implemented location-aware product filtering at the PostgREST query layer, scoping marketplace listings to the buyer's city without additional infrastructure.

Person 5 Testing, Quality & Risk Analysis

Anas Ibarim

QA Lead & Risk Analyst

Concept Definition

Testing, Quality & Risk Analysis encompasses the systematic verification that a software system meets its requirements, the identification of failure modes before they reach users, and the mitigation of risks through layered defensive engineering. In Usaruna this included both automated test suites and database-level safeguards against business-critical failure scenarios.

Key Contributions

  • Vitest Unit Test Suite

    Authored unit tests with Vitest for all pure utility functions: cart merge logic, parseRpcError(), price calculation helpers, and form validators.

  • Playwright E2E Tests

    End-to-end test scenarios covering the complete purchase journey: browse → add to cart → login → merge → checkout → order confirmation, and the seller product management flow.

  • Risk: Overselling Mitigation

    Documented and verified the ACID place_order RPC as the mitigation for the overselling race-condition risk — stock decrement and order creation are atomic.

  • Risk: Unauthorized Access Mitigation

    Verified RLS policies through integration tests that assert cross-user data access returns empty sets, confirming that no seller can read another's orders or products.
